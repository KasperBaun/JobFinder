## What was built but NEVER run end-to-end

**LLM judging — Phase 1 (`a38c721`) and Phase 2 (`5e46fff`).**

The `Jobmatch.Llm` namespace ships an `ILlmClient` interface with two
implementations — `LlamaSharpClient` (in-process via
LLamaSharp NuGet, default) and `OllamaClient` (HTTP, opt-in for power
users). `LlmJudge` builds a few-shot prompt from skillset +
`data/<email>/examples/*.md` and asks for `{"score":0-1,"reason":"…"}`
per match. `SearchService` calls it between Ranker.Score and the
filter step when `ranking.Llm.Enabled` is true, blends
`weight*llm + (1-weight)*keyword`, re-sorts.

Phase 2 added `GET /api/llm/status` + `POST /api/llm/download-model`
(SSE-streaming GGUF download from a configurable Hugging Face URL)
and a `<LlmModelBanner />` on the Search page. The default download
target is mradermacher's Gemma 3 4B Q4_K_M (~2.32 GB, public, no auth).

**It compiles and tests pass — but no human has flipped

  has a strict + loose fallback, but Gemma 3 4B may need prompt
  tuning. Iterate on `BuildSystemPrompt` in
  `src/backend/Jobmatch/Llm/LlmJudge.cs`.
- **Each call too slow on CPU**: lower `llm.top_n` from 50 to 20 in
  ranking.yml; 20 listings × 5 sec = 100 sec, much more interactive.
