## Your first action — fix the LlmModelDownloader SSL bug

This is the **P0**. Phase 2's "no manual GGUF placement" promise is
currently broken end-to-end. I worked around it via curl, but any
new user hitting the GUI banner gets a download failure on first
click.

**Symptom**
```
POST /api/llm/download-model
→ SSE event: {"type":"error","message":"The SSL connection could not be established, see inner exception."}
```

**Repro** (assumes no model file exists)
```
# delete the existing model first to repro
rm "data/kasperbaun@hotmail.com/models/gemma-3-4b-it-q4_k_m.gguf"
# enable LLM (already true), start host, hit endpoint
curl -sN -X POST http://127.0.0.1:5183/api/llm/download-model
```

**Root cause**
HuggingFace's first hop (`huggingface.co`) requests TLS renegotiation
during the HTTPS connection — curl's SChannel handles it
(`schannel: remote party requests renegotiation` in `-v` output),
.NET's `SocketsHttpHandler` (the default behind `HttpClient`) does
not allow renegotiation by default. The inner exception that would
say so was suppressed because `Program.cs:22-23` clears all logging
providers.

**Recommended fix**
In `src/backend/Jobmatch.Api/JobmatchApiExtensions.cs:40`, replace:

```csharp
services.AddHttpClient<LlmModelDownloader>(c => c.Timeout = TimeSpan.FromHours(2));
```

with:

```csharp
services
    .AddHttpClient<LlmModelDownloader>(c => c.Timeout = TimeSpan.FromHours(2))
    .ConfigurePrimaryHttpMessageHandler(() => new SocketsHttpHandler
    {
        SslOptions = new System.Net.Security.SslClientAuthenticationOptions
        {
            AllowRenegotiation = true,
        },
    });
```

Verify by deleting the model file, restarting, and watching the
download complete via SSE. Confirm the GUI banner clears.

**Bonus while you're in there**: capture `ex.InnerException?.Message`
in `LlmHandler.DownloadModel`'s catch block so future SSL bugs
surface diagnostically instead of as the opaque outer message.

## Next, in priority order

### P1 — Host logging is fully suppressed (debug rage-fix)

**Symptom**
When the LLM silently fell back to keyword-only today, host stdout
said nothing. I had to temporarily edit `Program.cs` to add a console
logger, repro, then revert. Every silent-failure bug currently
requires this dance.

**Root cause**
`Program.cs:22-23` clears all logging providers and adds a
`LogLevel.None` filter — by design ("the terminal belongs to the
user"). But it also drops WARN / ERROR that signal real bugs.

**Recommended fix**
Keep the terminal quiet, but log Warning+ to a file at
`data/<email>/logs/host.log` (single rotating file is fine — bound
to a few MB). Surface the path in the `/api/system/ping` response or
the GUI footer so users know where to find it.

Roughly:
```csharp
builder.Logging.ClearProviders();
builder.Logging.AddFile($"{ctx.RootDir}/logs/host.log", LogLevel.Warning);
```
(Pick an `ILoggerProvider` impl — `Serilog.Sinks.File` or
`NReco.Logging.File`. Both are <100KB packages.)

### P1 — Cross-portal duplicates still flood the top 10

**Symptom**
Top 10 has Sopra Steria × 2 (#1, #2), Danske Spil × 2 (#4, #5),
DR Teknologi × 2 (#9, #10). Half the top 10 is duplicates of the
same job listed on two portals.

**Root cause**
`Deduper.Deduplicate` collapses on URL OR `(normalized_title |
company | location)`. The Jobindex RSS feed appends `, <company>`
to the title:
- Jobindex: `"Senior .Net udvikler til afdeling i vækst, Sopra Steria A/S"`
- SmartRecruiters: `"Senior .Net udvikler til afdeling i vækst"`

The trailing `, <company>` defeats the title hash.

**Recommended fix**
In `Deduper.NormalizeTitle` (or wherever titles get normalized
before hashing), strip a trailing `, <CompanyName>` suffix when the
company is known. The previous handoff already flagged this exact
issue under "Tighten dedupe".

### P2 — Source-specific remote-mode extraction

**Symptom**
5 of the top 10 shortlist still tag `Unknown` (Sopra Steria smartrec,
CHANGE, Unity, both DR Teknologi). The frontend now hides those
badges, but the data is recoverable.

**Root cause**
`InferRemoteMode` scans `{title} {description} {location}` for
keywords. SmartRecruiters' JSON API has a `customField` array
that often holds "Workplace policy / Hybrid" — we don't read it.
hr-manager's JSON-LD scrape often has a `workLocation` or
`employmentType` field — we don't read it.

**Recommended fix**
Per-adapter: extract a structured remote-mode from the source
fields where available, and only fall through to `InferRemoteMode`
when those fields are silent. Two cheap wins:
- `ApiAdapter` (SmartRecruiters): parse `customField` looking for a
  field matching `name=~/workplace|policy|remote/i`.
- `HrManagerAdapter`: read `workLocation.@type` from the JSON-LD.

### P2 — LLM judging is slow on CPU (~19 sec/listing)

The handoff estimated 5 sec/listing on a "Ryzen / M2-class laptop".
Reality on this machine is ~19 sec/listing → 50 listings ≈ 16 min.

Things to try, in order of ROI:
1. **System-prompt KV caching** (biggest win, ~5-10×). The system
   prompt is identical across all 50 calls — only the user prompt
   varies. Pre-tokenize the system prompt into a "warm" KV state
   once, then re-evaluate just the user prompt against it. Requires
   reworking `LlamaSharpClient.ChatAsync` to keep a snapshot of the
   context immediately after system-prompt ingestion and restore from
   it instead of full `MemoryClear` between calls. See
   `LLamaContext.SaveState` / `LoadState` in LLamaSharp 0.27.
2. **GPU offload**. Most users have a discrete GPU. Add a
   `Cuda12` / `Vulkan` backend swap option (already a documented
   knob in `ranking.yml: llm.gpu_layer_count`). Currently the user
   would have to swap `LLamaSharp.Backend.Cpu` for
   `LLamaSharp.Backend.Cuda12` in `Directory.Packages.props`.
3. Lower `llm.top_n` from 50 → 25 in `ranking.yml` (halves wall
   time; risks missing marginal listings the LLM would re-rank up).
