## Recent

- Slow sources no longer hold up the whole run (R-089/R-099) — fetching was fully concurrent but gated on `Task.WhenAll` with no per-source timeout and a shared `HttpClient` on the framework-default 100s per-request timeout, so the highest-volume enrich-body sources (`jobindex-rss-software-engineer` ~160 items ×2 fetches, `workday-simcorp` ~280) ran 3+ min doing strictly-sequential body enrichment (200ms + up to two live third-party fetches each) and every other source waited on them. Now: fetching uses a dedicated client with a 30s per-request timeout (the judge keeps its own untimed client — Ollama generations exceed 30s), each source has a 120s end-to-end budget (`FetchSafe` linked CTS) after which it's abandoned and surfaced as `… timed out after 120s` while the run proceeds, and body enrichment runs bounded-concurrent (degree 4, order-preserving) so legit high-volume feeds finish in well under a minute with full data. Enrichment split into `BaseAdapter.Enrichment.cs`. The three chronically-failing sources (`recruit-it` 454, `html-jyskebank` 403, `jobnet` HTML login page) are WAF/paywall blocks (anti-bot bypass forbidden) — disable them in the Sources page.
- Application-status tracking + outcome calibration (R-096/R-097/R-098) — listings take an applied/interview/offer/rejected/no-response status independent of the good/bad mark (`marks.json` entries persist while either is set; legacy shapes still read), a new cross-run Applications view (`GET /api/applications`, newest run wins per listing), and interview/offer outcomes join the judge's few-shot examples as liked signal that outranks plain marks. *(Concept: ai-job-search `job_search_tracker.csv`.)*
- Mark reasons close the feedback loop (R-094/R-095) — a set mark grows a "+ why?" affordance (inline input, editable chip; reason drops when the mark flips or clears; `marks.json` stores `{mark, reason}` objects, bare strings still read). On the next run the judge's few-shot block now includes listings marked in previous runs (newest opinion per title+company wins, curated `examples/` win collisions) *and* the free-form body of curated example files, which `ToFewShotPrompt` used to drop — each rendered with "— why: <reason>"; note-bearing examples survive the cap first. So "AI Engineer - Student" at 0.81 is fixed by marking it bad with "I'm not a student" once.
- CV-driven profile setup (R-011) — "Fill from CV" on the Profile page (and a link in the first-run wizard) runs the local Gemma model over a pasted/uploaded/URL-fetched CV (`PdfPig` for PDFs), shows a per-field current→suggested diff, and prefills only the accepted fields; nothing persists until the user saves (R-012). Extraction runs on a `CvExtractionManager` singleton (`ModelDownloadManager` pattern) so it survives navigation/reload; `POST /api/skillset/extract` + status poll. Concept adopted from MadsLorentzen/ai-job-search, which also seeded five new backlog items (see todo.md).
- Search progress no longer over-counts sources — the live "N/M" total and source grid seeded from every *user-enabled* source, but the run actually fetches only sources whose required secret is set (the backend's `Prepare`/`ProviderStateMerger` filter). So a key-less source (e.g. careerjet-dk `affid`, jooble `api_key`) inflated the total and sat "pending" forever — the grid showed 45/50 while the floating pill correctly showed 45/48. The frontend now mirrors that filter (`enabled && (!requiresSecret || hasSecret)`) when seeding the run set, and the source picker greys out secret-less sources with a "Needs an API key" hint instead of pretending they'll run. The Sources page now names the trap directly: a key-less enabled source gets a first-class amber **"needs key"** health state ("won't run until keyed" + an **Add <key> →** CTA) instead of reading as OK/stale — so "On but can't run" is a visible state, not a silent surprise.
- Desktop "Close jobfinder" quits in one click — inside the Electron shell the quit button now closes the native window and exits the app (same graceful path as the titlebar X: `before-quit` stops the backend, then `app.exit(0)`), instead of the browser-shell flow of confirm → stop backend → dead-end "Goodbye, close this tab" screen. A new preload bridge (`window.jobfinderDesktop.quit()` → `ipcMain 'jobfinder:quit'`) also lets the SPA tell it's running in the desktop app; the browser web-shell keeps the confirm + goodbye page (there a tab really can't close itself). No confirm in desktop — a stray quit is cheap since searches run as durable background jobs.
- Compact live search-progress view — the running-search card no longer scrolls for pages. A numbered pipeline stepper (① Retrieve jobs → ② Deduplicate → ③ Rate matches → ④ AI review → ⑤ Done, AI step shown only when the model is present) leads the card, followed by a compact summary (sources progress bar + live elapsed timer + "N jobs found · K running · M failed") and a dense responsive **source grid** (bounded, scrolls internally) replacing the 48-row list. The redundant per-event timeline moved into a collapsed "Activity log" disclosure. The progress view was extracted from `SearchPage.tsx` (301→209 lines) into `SearchProgress`/`PipelineSteps`/`ProgressSummary`/`SourceGrid`/`ActivityLog` + a `useElapsed` hook; no backend/SSE change.
- Model download survives navigation + reload (R-093) — the Gemma download used to be a request-bound SSE stream held in the banner's local state, so navigating away lost all track of it (and a re-click blocked forever on the still-held download lock, stuck at "Downloading 0.0 MB"). It now runs on a process-singleton `ModelDownloadManager` decoupled from the request; live progress is exposed on `GET /api/llm/status`, which the banner polls while downloading and reconnects to on remount. Start is idempotent (a repeat click is a no-op), and a partial `.download` temp is promoted only on completion. `POST /api/llm/download-model` returns immediately instead of streaming SSE.
- Linux desktop packaging (`.deb`) — `npm run package:linux` publishes the self-contained `linux-x64` backend and builds an electron-builder `.deb` (installs to `/opt/Jobfinder`, app-menu entry, `apt`-managed). The backend was already cross-platform; the change parameterises the publish RID in `publish-backend.mjs`, moves `extraResources` into per-OS blocks in `electron-builder.yml` (adds a `linux`/deb target + `homepage`), and fixes the dev-mode publish path in `paths.ts`. A custom deb post-install forces the SUID `chrome-sandbox` (the default `unshare` probe runs as root at install time and mis-detects, so the app crashed with SIGTRAP on Ubuntu 24.04 where apparmor blocks the userns sandbox), and `make-icons.mjs` now emits a `build/icons/` hicolor set so the app icon actually shows (a lone 1024px png is ignored by GNOME). Windows/`package:win` behaviour unchanged.
- The Windows installer is now the Electron desktop app — `npm run package:win` builds the electron-builder NSIS installer (was the Inno Setup browser installer); the Inno recipe + `package-installer.mjs` are retired. The browser experience stays functional via the `jobfinder` .NET tool and `npm run dev`, just without its own installer.
- Native Electron desktop shell + app icons — wraps the bundled backend in a native window instead of a console + browser tab: the main process picks a free loopback port, spawns `Jobmatch.Host.exe` hidden (`JOBFINDER_PORT` + `JOBFINDER_NO_BROWSER=1`), polls `/api/system/ping`, then opens the SPA in a `BrowserWindow`. Layered shutdown so the backend is never orphaned, single-instance lock, startup-error window (now carrying the app icon); electron-builder NSIS installer is the Windows desktop distribution. Source now tracked under `src/desktop/`. (R-092)
- Desktop window fits the screen + remembers its bounds — launch size adapts to the display's work area (never larger than the screen, so the main view isn't clipped on scaled/high-DPI displays), and size/position/maximized state persist across runs (`window-state.json`); on a short work area the effective minimum shrinks to fit. Wide result tables (the 8-column longlist) now scroll horizontally instead of clipping.
